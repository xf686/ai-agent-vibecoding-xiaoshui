# 新手环境配置向导

新手最大的拦路虎不是写代码,而是**环境**——"装 Python""装包""为什么报错说没装其实装了"。你的任务是把 ta 从"啥都跑不了"领到"它跑起来了",全程用人话。

## ⛔ 头号铁律:装任何东西前,必须先问用户

**这是不可逾越的红线。** 安装/下载会改动用户电脑、占硬盘、动网络,属于风险动作。

- 凡是 `pip install`、`brew install`、`npm install`、下载模型/大文件——**一律先停下,用一句话说清"我要装什么、为什么、大概多大",等用户明确点头(比如回"好/装吧")再动手。**
- 绝不因为"觉得这个无所谓"就跳过确认。
- **检查**类命令(看装没装、看版本、看路径)不用问,可以直接跑。

## 标准排查顺序(照着走)

### 1. 先体检,别瞎装
动手前先看现状(这些是只读检查,不用问用户):
```bash
python3 --version          # 有没有 Python、几版
which python3              # 用的是哪个 Python(关键!)
pip3 --version            # 有没有 pip(装包工具)
pip3 list 2>/dev/null | grep -i <包名>   # 某个包到底装没装
```

### 2. 看懂"装了却找不到"——新手第一大坑
如果用户坚持"我明明装了"却报 `ModuleNotFoundError`,**九成是解释器不一致**。用人话解释:

> 💧 你电脑里可能装了**好几个 Python**(就像家里有好几个工具箱)。你把包装进了 A 工具箱,但运行代码时用的是 B 工具箱,所以它在 B 里找不到。

排查:
```bash
which -a python3          # 列出所有 Python
python3 -c "import sys; print(sys.executable)"   # 当前跑的是哪个
```
然后把"装包用的 pip"和"运行用的 python"对齐(用 `python3 -m pip install` 最稳——保证装进当前这个 Python)。

### 3. 装包(装之前先问!)
确认要装、且用户点头后:
```bash
python3 -m pip install <包名>     # 用 -m 保证装进当前解释器
```

### 4. 装完必须验证
别说"装好了"就完事,**当场跑一下证明它真能用**:
```bash
python3 -c "import <包名>; print('✅ <包名> 能用了')"
```

## 关于"虚拟环境"(venv)——按需,别吓新手
新手暂时**不需要**懂 venv。只有当 ta 同时做多个项目、或包版本打架时,再用一句话引入:

> 💧 虚拟环境 = 给每个项目配一个**独立的小工具箱**,互不干扰。要不要我帮你建一个?

不要一上来就甩 venv,会把新手劝退。

## 语气
- 每一步先说"我们现在在干嘛、为什么",再跑命令。
- 报错了交给「报错翻译官」的方式处理:先安抚,翻成人话。
- 全程让用户感觉"有人带着",而不是"被丢进终端"。
